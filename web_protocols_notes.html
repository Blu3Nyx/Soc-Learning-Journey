Chapter 01

What is a Protocol? 🤝

Think of it like a language computers use to talk to each other.

A web protocol is a set of rules that defines how data is sent and received over a network. Just like humans need a common language to talk — computers need protocols to communicate.
👤
Humans
Use languages like English, Hindi to communicate
💻
Computers
Use protocols like HTTP, DNS, SMTP to communicate
🚨 Without protocols — devices would not understand each other at all. It would be complete chaos!

Where do protocols live? (OSI Model) 🧱

Different protocols work at different "layers" — like floors in a building. Each floor has a specific job.
LayerWhat it doesProtocols
ApplicationWhat you directly use — apps, browsersHTTP, HTTPS, DNS, FTP
TransportMakes sure data arrives correctlyTCP, UDP
NetworkFinds the path to the destinationIP
Data LinkLocal delivery between nearby devicesEthernet, WiFi

HTTP 🌍
HyperText Transfer Protocol

The foundation of every website you've ever visited. Simple, but powerful — and a bit insecure.

HTTP is the core protocol your browser uses to load websites. When you type a URL and press enter — HTTP takes over.
80
Port Number
🔓
Not Encrypted

Stateless
Forgets you every time
🔄 How HTTP Works — Step by Step
📄
Your Browser Web Server
1
You type http://example.com in your browser and press Enter
2
Browser sends a request message to the server: GET /index.html HTTP/1.1
3
Server receives the request and finds the file you asked for on its hard drive
4
Server sends back a response: HTTP/1.1 200 OK with the full HTML page
5
Your browser reads the HTML and displays the webpage on your screen

Different ways you can "talk" to a server — each method has a specific job.

⚠️ Stateless — Very Important Concept!
HTTP does NOT remember you between requests. Every single request is treated as brand new — like you're a stranger every time!
❌ HTTP Forgets:
Who you are
Your previous requests
Your login status
✅ Solution — We use:
🍪 Cookies (stored in browser)
📋 Sessions (stored on server)
🎫 Tokens (JWT — sent with each request)

HTTP + Encryption = HTTPS

HTTP vs HTTPS — The Difference
HTTP — Not Encrypted 😨
📄
Your Browser Server
⚠️ Anyone on the network can READ this data (passwords, card numbers — everything!)
HTTPS — Fully Encrypted 🔐
🔒
Your Browser Server
✅ Data is scrambled — even if intercepted, no one can read it!
HTTPS Provides 🛡️
✅ Confidentiality — No one can read your data
✅ Integrity — Data can't be secretly changed
✅ Authentication — You're on the real website
🔌 Port: 443
🔐 Uses: SSL / TLS encryption
Without HTTPS 😱
❌ MITM Attack possible — attacker sits in the middle reading everything
❌ Your passwords can be stolen
❌ Credit card numbers visible
❌ All data in plain readable text

DNS 📖
Domain Name System — The Internet's Phonebook

Computers only understand IP addresses (like 142.250.1.1) — but humans prefer names (like google.com). DNS is the translator that converts names → numbers.
google.com ──── DNS ────▶ 142.250.x.x
53
Port Number
UDP
Transport (mostly)
🔍 How DNS Resolves "google.com"
1
Browser Cache — "Have I visited google.com recently?" If yes → uses saved answer. Fastest!
2
OS Cache — Checks Windows/Mac's own stored DNS answers
3
ISP Resolver — Asks your internet provider (Jio/Airtel) "Do you know google.com?"
4
Root DNS Server — Top of the hierarchy. Says "I don't know but ask the .com server"
5
TLD Server — The .com server says "Ask Google's own server"
6
Authoritative Server — Google's own DNS server gives the final IP address ✅

Iterative vs Recursive DNS Query 🔄

🔁 Iterative Query
Your local DNS server does all the work itself. It asks Root → gets a hint → asks TLD → gets another hint → asks Authoritative → gets the answer. It makes multiple trips on its own.
♻️ Recursive Query
Your local DNS server asks Root and says "YOU find it for me". Root passes to TLD, TLD passes to Authoritative — all on your behalf. Final answer comes in one go.
Chapter 05

More Key Protocols ⚡

FTP, SFTP, SMTP, SSH, RDP, SMB — what each one does, their ports, and what attackers do with them.

📁
FTP / SFTP
File Transfer Protocol
21 / 22

Used to move files between computers — like uploading files to a web server or downloading a backup.

FTP — Port 21
❌ NOT secure
Sends everything in plain text
Has Active & Passive modes
SFTP — Port 22
✅ Runs over SSH
Fully encrypted
Always prefer this!
⚠️ Large file transfers ⚠️ Anonymous logins 💀 Credential sniffing 💀 Data exfiltration
📧
SMTP
Simple Mail Transfer Protocol — Email Sending
25 / 587

SMTP sends emails. When you hit "Send" on Gmail, SMTP carries your email from your device → through the internet → to the recipient's mail server.

📱 Your Device → SMTP → ☁️ Internet → SMTP → 📬 Mail Server → POP3/IMAP → 👤 Recipient
⚠️ Bulk email sending ⚠️ Email spoofing 💀 Phishing 💀 Spam campaigns
💻
SSH — Secure Shell
Securely control a remote computer via terminal
22

SSH lets you remotely login to another computer and type commands into it — completely securely. Like a sysadmin controlling a Linux server from thousands of miles away.

⚠️ Multiple failed logins ⚠️ Login from unknown IP ⚠️ Root login attempts 💀 Brute force 💀 Credential theft
🖥️
RDP — Remote Desktop
Full graphical control of a Windows PC remotely
3389

RDP gives you a full Windows desktop view of a remote computer — as if you were physically sitting in front of it. Used a lot in corporate IT support.

🚨 Major Attack Target! RDP exposed to internet = guaranteed brute force attempts. Attackers scan for open port 3389 constantly.
⚠️ Login at odd hours ⚠️ External IP logins 💀 RDP Brute Force 💀 Lateral Movement 💀 Ransomware Entry
📂
SMB — Server Message Block
Windows file sharing on local networks
445

SMB lets Windows computers share files, printers, and resources with each other on a local network. Those shared drives you see in File Explorer? SMB is behind that.

⚠️ Access to C$ ADMIN$ shares ⚠️ File movement between systems 💀 EternalBlue Exploit (WannaCry) 💀 SMB Relay Attack 💀 Lateral Movement
Chapter 06

Computer Ports 🔌
65,535 doors — each one for a different purpose

A port is like a numbered door on a computer. The IP address gets you to the building — the port number tells you which apartment to knock on. This way, your computer can handle email, web browsing, file transfers — all at the same time!
0 – 1023
Well-Known Ports
Reserved for standard universal services that never change. HTTP (80), HTTPS (443), DNS (53), SSH (22), FTP (21). Everyone uses these exact ports.
1024 – 49151
Registered Ports
Used by specific applications — officially registered but not universal. RDP (3389), MySQL (3306), SQL Server (1433). Software picks a specific port here.
49152 – 65535
Dynamic Ports
Temporary ports — your browser picks one randomly each time you make a connection. Used and then released. Totally normal to see these in logs.

📋 Must-Know Ports — Memorize These!

Chapter 07

Real SOC Log Example 🔍
Think like a SOC analyst

This is what a real suspicious log looks like. Let's break it down together.

Source IP : 185.221.x.x← External IP (from internet)
Destination IP : 192.168.1.10← Internal machine
Port : 3389← That's RDP!
Attempts : 500← 500 tries!
Status : FAILED← All failed
🔍
Port 3389? → Immediately think: RDP — someone wants FULL remote desktop access to a Windows machine
🔍
Source IP 185.221.x.x? → This is a PUBLIC IP — attacker is coming from the open internet, not inside the company
🔍
500 attempts, all FAILED? → Nobody accidentally tries 500 times. This is 100% a brute force attack — automated tool trying thousands of passwords
🔍
Destination 192.168.1.10? → Private internal IP — the attacker is targeting a specific machine inside the company network
🚨 Conclusion: RDP Brute Force Attack!

An external attacker is using automated tools to try 500 different passwords on port 3389 (RDP) to gain full remote desktop access to an internal Windows machine. Immediate action needed: Block the source IP at the firewall, check if any attempt succeeded, alert the security team, and check why port 3389 is exposed to the internet at all!
